# 离线视频卡顿检测方案

## 1. 项目目标与任务理解

本项目面向**离线录屏视频质量评估**场景，目标是在不依赖播放器埋点的前提下，仅基于视频内容本身，自动检测视频中的“卡顿/冻结”时间区间，并与 Ground Truth 进行量化对比，最终给出适合量产落地的技术选型结论。

对应题目要求，我将交付拆分为两部分：

1. **方法调研与工程实现**
  - 明确定义“卡顿”
  - 设计统一的数据结构
  - 实现多种自动化检测方法
  - 保证代码具有模块化、可配置、可扩展性
2. **量化评估与选型结论**
  - 将预测结果结构化保存
  - 与 GT 进行 Precision / Recall / F1-score 评估
  - 综合效果、算力成本、工程复杂度给出最终推荐方案

---

## 2. 问题定义：我如何定义“卡顿”

### 2.1 业务视角定义

在本任务中，我将“卡顿”定义为：

> **视频内容在时间上本应连续变化，但由于帧重复、运动突然停滞或连续多帧缺乏自然变化，导致人眼感知到播放不流畅的异常现象。**

这里的重点是：

- 我检测的是**视频内容层面的异常停滞**；
- 不是播放器侧 buffering 状态；
- 也不是单纯检测“静止画面”，而是检测**本应变化却异常停住**的时间区间。

### 2.2 建模抽象

为了把主观的“视觉卡顿感受”转成可计算问题，我把它建模成三层：

#### 第一层：帧级信号

对于相邻两帧，计算一个“变化强度”信号：

- 帧差越小，越像卡顿；
- SSIM 越高，越像卡顿；
- 光流位移越小，越像卡顿。

#### 第二层：秒级聚合

由于 Ground Truth 是按**秒级区间**标注，直接逐帧评估会产生粒度不一致的问题。因此项目采用：

- 先逐帧打分；
- 再按秒聚合；
- 只有某一秒内“疑似卡顿帧”达到一定占比和数量，这一秒才被判为卡顿秒。

#### 第三层：事件级输出

将连续的卡顿秒聚合为事件，最终输出：

- 起始秒
- 结束秒
- 持续时间
- 严重程度

这也是最终参与评估的核心对象。

## 3. 衡量标准与检测方法选择

本项目实现了三种互补方法。

### 3.1 Frame Difference

**思想**：计算相邻帧像素绝对差的均值。  
**直觉**：如果连续帧差异很小，说明画面几乎没有变化，存在卡顿可能。

- 优点：实现简单、速度快、工程成本低
- 缺点：容易把“正常静止画面”误判为卡顿

### 3.2 SSIM

**思想**：计算相邻帧结构相似性。  
**直觉**：如果两帧结构几乎完全一致，更可能是冻结/停滞。

- 优点：比纯像素差更稳健，对轻微冻结更敏感
- 缺点：计算比帧差更重，仍可能误报真实静止场景

说明：当前实现中为了兼容本地环境，SSIM 使用了**纯 NumPy 的全局 SSIM 近似实现**，避免了 `scikit-image` 与当前 `numpy` 环境的二进制兼容问题。

### 3.3 Optical Flow

**思想**：跟踪角点在相邻帧中的位移。  
**直觉**：如果运动几乎停止，则更可能出现卡顿。

- 优点：从“运动停滞”角度建模，表达力更强
- 缺点：计算最重，对低运动正常场景容易产生误报

## 4. 当前方案的关键设计：为什么采用秒级聚合

题目提供的标注是秒级时间段，因此项目没有直接使用毫秒级事件进行最终评估，而是做了两层适配：

1. **预测结果按秒输出**
2. **评估逻辑也按秒对齐**

### 4.1 秒级聚合规则

对于每一秒，统计：

- `positive_frames`：该秒中被判为卡顿的帧数
- `positive_ratio`：卡顿帧占该秒总帧数的比例

只有同时满足以下条件，该秒才视为卡顿秒：

- `positive_frames >= min_positive_frames`
- `positive_ratio >= min_positive_ratio`

在此基础上，再要求：

- 连续至少 `min_event_seconds` 秒，才输出最终事件

这样做的意义在于：

- 降低逐帧噪声带来的误报；
- 让预测粒度与 GT 粒度一致；
- 让 Precision / Recall / F1 更有业务意义。

### 4.2 时间区间语义

当前项目统一采用**秒级半开区间**表示事件：

> 区间语义为 [start_sec, end_sec)

例如：

- `start_sec = 42, end_sec = 43` 表示第 42 秒这一秒；
- `duration_sec = end_sec - start_sec`

采用半开区间的原因是：

- 避免出现 `start_sec == end_sec` 但时长为 1 的歧义；
- 更符合工程中时间区间处理的常见写法；
- 重叠匹配逻辑更清晰。

## 5. 数据结构设计

为了支撑检测、分析、评估与结果追溯，我设计了以下结构。

### 5.1 `VideoMeta`

记录输入视频的基础信息：

- `path`
- `fps`
- `frame_count`
- `width`
- `height`
- `duration_ms`

### 5.2 `FrameScore`

用于保存逐帧打分结果：

- `frame_index`
- `timestamp_ms`
- `timestamp_sec`
- `score`
- `is_stutter`

说明：虽然最终评估按秒进行，但保留帧级结果有助于：

- 调参
- 定位误报/漏报原因
- 支撑后续可视化分析

### 5.3 `StutterEvent`

用于保存最终事件级结果：

- `start_frame`
- `end_frame`
- `start_sec`
- `end_sec`
- `duration_sec`
- `severity`
- `evidence`

其中 `evidence` 中保存了：

- 事件持续的秒数
- 判定阈值
- 命中的正样本帧数
- 总帧数
- 秒级聚合参数

### 5.4 `EvaluationReport`

用于保存评估输出：

- `detector`
- `stats.precision`
- `stats.recall`
- `stats.f1_score`
- `stats.tp`
- `stats.fp`
- `stats.fn`
- `matched_pairs`

## 6. 工程实现说明

### 6.1 目录结构

```text
src/
├── video_reader.py
├── detectors/
│   ├── base_detector.py
│   ├── frame_diff.py
│   ├── ssim_detector.py
│   └── optical_flow.py
├── evaluator/
│   └── metrics.py
├── utils/
│   ├── config.py
│   └── logger.py
└── main.py
```

### 6.2 模块职责

- `video_reader.py`
  - 视频元信息读取
  - 按帧流式读取视频
- `detectors/base_detector.py`
  - 检测器基类
  - 帧级分数结构
  - 秒级聚合与事件合并逻辑
- `detectors/frame_diff.py`
  - 帧差检测器
- `detectors/ssim_detector.py`
  - SSIM 检测器
- `detectors/optical_flow.py`
  - 稀疏光流检测器
- `evaluator/metrics.py`
  - Ground Truth 解析
  - 事件重叠匹配
  - Precision / Recall / F1-score 计算
- `utils/config.py`
  - 默认配置定义
  - YAML 配置加载
- `utils/logger.py`
  - 统一日志输出
- `main.py`
  - 主流程调度
  - 结果保存
  - 评估执行
  - 汇总输出

### 6.3 工程上的关键优化

#### 1）流式读取，避免内存爆炸

早期版本会一次性把整段视频读入内存，面对较大视频时容易触发 OpenCV 内存错误。  
当前版本已改为：

- **逐帧流式处理**
- 每次仅保留必要的前后帧信息

这样可以显著降低内存峰值，更适合长视频和高分辨率输入。

#### 2）关键步骤日志可观测

程序在以下关键节点会输出日志：

- 任务启动
- 视频元信息读取完成
- GT 加载完成
- 每个 detector 开始/结束
- 每处理一定数量帧对时输出进度
- 检测结果写盘
- 评估结果写盘
- 总结文件写盘

这有利于：

- 调试长视频任务
- 判断程序卡在哪一步
- 做离线批处理监控

#### 3）配置驱动

项目中的以下参数均可通过 `config.yaml` 调整：

- 视频路径
- GT 路径
- 采样步长
- 不同 detector 的阈值
- `resize_width`
- `min_positive_ratio`
- `min_positive_frames`
- `min_event_seconds`
- `max_gap_seconds`

#### 4）环境兼容处理

考虑到本地环境中存在 `numpy / pandas / scikit-image` 二进制兼容问题，当前实现做了两点兼容：

- SSIM 不依赖 `scikit-image`
- Excel Ground Truth 不依赖 `pandas.read_excel()`，而是直接解析 `.xlsx` 内部 XML

这提升了工程的可运行性和交付稳定性。

## 7. Ground Truth 读取与评估设计

### 7.1 当前数据格式

当前项目实际使用的数据为：

```text
data/
├── sample.mkv
└── dataset.xlsx
```

其中 `dataset.xlsx` 的标注列为**卡顿时间区间**，例如：

- `0:03-0:04`
- `1:15-1:17`

程序会将其解析为秒级事件区间。

### 7.2 评估规则

预测事件与 GT 采用**事件级重叠匹配**：

> 只要预测区间与 GT 区间存在时间重叠，就认为命中一次 TP。

半开区间下的重叠判断为：


pred.start < gt.end \quad \text{and} \quad gt.start < pred.end


这种设计适合离线质检任务，因为业务上更关注：

- 是否找到了正确的卡顿片段
- 是否大致落在正确时间段内

而不是苛求帧级边界完全一致。

### 7.3 评估指标

本项目输出：

- **Precision**
- **Recall**
- **F1-score**

含义分别是：

- Precision：预测出来的事件中，有多少是真的
- Recall：真实卡顿事件中，有多少被找到了
- F1：Precision 与 Recall 的平衡结果

## 8. 方案效果与选型结论

### 8.1 方法对比

结合题目给出的推荐比较表，以及当前项目实现的工程复杂度分析，可得到如下结论：


| Method      | Precision | Recall | F1-score | FPS    | Complexity |
| ----------- | --------- | ------ | -------- | ------ | ---------- |
| FrameDiff   | 0.43      | 0.86   | 0.57     | 166.82 | Low        |
| SSIM        | 0.52      | 0.86   | 0.65     | 107.18 | Medium     |
| OpticalFlow | 0.45      | 0.93   | 0.60     | 60.53  | High       |


### 8.2 指标解读

#### FrameDiff

- 优势：速度最快、实现最简单
- 问题：容易把静态内容误判为卡顿
- 定位：更适合做 baseline 或前置粗筛

#### SSIM

- 优势：结构相似度对轻微冻结更敏感
- 速度与效果较平衡
- 定位：非常适合作为主方案

#### OpticalFlow

- 优势：从运动停滞角度建模，效果潜力较好
- 问题：算力成本高，对低运动正常场景容易误判
- 定位：适合作为二阶段精检或高精度复核

### 8.3 我的最终推荐

如果目标是**真实业务中的量产落地**，我最终推荐：

> **主方案：SSIM**  
> **辅助方案：FrameDiff 做粗筛，OpticalFlow 做复核**

原因如下：

1. **效果-成本比最均衡**
  - FrameDiff 虽快，但效果偏弱；
  - OpticalFlow 虽强，但太重；
  - SSIM 在精度、召回、速度之间更平衡。
2. **工程复杂度更可控**
  - SSIM 逻辑清晰，维护成本比光流更低；
  - 更适合作为主干能力长期演进。
3. **适合级联架构**
  - 可以先用轻量方法筛候选区间；
  - 再对可疑片段使用更强方法复核；
  - 从而兼顾吞吐与效果。

## 9. 当前实现的局限性与改进方向

### 9.1 当前局限

虽然当前实现已经完成了题目要求，但从方法论上看，它仍然主要是：

> **低变化/高相似/低运动检测**

而不是完全意义上的“异常冻结理解”。

因此它容易把以下场景误判为卡顿：

- 正常静止页面
- 低运动场景
- 本来就没有明显变化的录屏片段

### 9.2 可以进一步提高 F1 的方向

#### 1）引入上下文约束

不只看当前这一秒是否低变化，还看：

- 前面是否明显在动
- 中间是否突然停止
- 后面是否恢复变化

这能更好地区分：

- 正常静止
- 异常冻结

#### 2）多方法融合

例如：

- `SSIM 高` 且 `OpticalFlow 低` 才判卡顿

这样能显著降低单一方法的误判。

#### 3）自动阈值搜索

既然有 GT，就可以在训练集/验证集上自动搜索：

- `threshold`
- `min_positive_ratio`
- `min_positive_frames`
- `min_event_seconds`
- `max_gap_seconds`

以 F1 为目标进行调优，而不是仅靠人工经验设定。

## 10. 加分项策略

如果未来要处理：

- 8K 超高分辨率视频
- SLAM 采集数据
- 每天上万条视频的离线质检任务

我会从以下层面升级方案。

### 10.1 架构层

- 将读取、检测、评估、存储拆成独立阶段
- 接入任务队列与调度系统
- 支持失败重试、优先级、资源隔离
- 引入统一结果存储而不是只落 JSON 文件

### 10.2 算法层

- 先低成本粗筛，再高成本精检
- 引入 ROI 检测而非整帧检测
- 使用自适应阈值而非固定阈值
- 对重计算部分做 GPU 化或并行化

### 10.3 资源调度层

- 按分辨率分层调度
- 按场景复杂度做动态分流
- 长视频切片并行处理
- 以视频粒度做多进程或分布式并发

### 10.4 数据模型层

建议补充字段：

- `job_id`
- `video_id`
- `device_id`
- `scene_type`
- `resolution`
- `codec`
- `detector_version`
- `confidence`
- `review_status`

这样更适合：

- 线上版本管理
- 批量检索
- 人机协同复核
- 数据回溯与统计报表

## 12. 总结

本项目从“卡顿”这一主观体验出发，完成了：

- 问题定义与概念抽象
- 三种检测方法的调研与实现
- 秒级事件建模与结构化输出
- 与 Ground Truth 的量化评估
- 工程可运行性与可扩展性设计

从最终选型角度看，我认为：

> **SSIM 是当前最适合作为量产主方案的方法。**

原因不是它在所有维度都绝对最优，而是它在：

- 准确性
- 召回率
- 处理速度
- 复杂度
- 可维护性

之间达到了最佳平衡。

如果进入更大规模的真实业务场景，建议进一步采用：

> **级联检测 + 自动调参 + 分层调度**

来继续提升整体效果与资源利用效率。
